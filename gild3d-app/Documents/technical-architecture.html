Executive Summary

Gilded Companions (gild3d.com) is a luxury private-booking platform connecting verified members with companion profiles. The platform provides profile browsing, secure messaging, calendar-based booking, and cryptocurrency payment processing — all within a privacy-focused, invitation-only environment.

Frontend
Next.js 14
Backend
Express + TS
Database
PostgreSQL 15
Hosting
AWS Lightsail

Key Design Principles

  • Privacy-first: No third-party analytics, private media gating, discreet billing references.
  • Container isolation: Every service runs in its own Docker container on a private bridge network.
  • Defence in depth: Security enforced at Cloudflare edge, Nginx reverse proxy, Express middleware, and database layers simultaneously.
  • Crypto-native payments: NOWPayments integration accepts BTC, LTC, ETH, USDC, DASH with no card-processor involvement.
  • Zero-downtime deploys: Docker Compose --force-recreate with health-check dependencies ensures services restart only when the DB is ready.

Cloudflare Real-IP Forwarding

Because all traffic passes through Cloudflare, the connecting IP seen by Nginx is always a Cloudflare IP. Nginx is configured with all 15 published Cloudflare IPv4 ranges under set_real_ip_from and uses real_ip_header CF-Connecting-IP to restore the true visitor IP. Express is configured with app.set('trust proxy', 1) for the same reason, ensuring rate limiting operates on real client IPs.

File Upload Security

  • Dual validation: File extension AND MIME type must both match the allow-list.
  • Allowed types: .jpg/.jpeg (image/jpeg) · .png (image/png) · .gif (image/gif) · .webp (image/webp) · .mp4 (video/mp4) · .mov (video/quicktime)
  • Random filenames: crypto.randomBytes(16).toString('hex') + ext — prevents enumeration, path traversal, and filename collision.
  • Size limit: 20 MB per file · 1 file per request.
  • Directory listing: Disabled via Nginx autoindex off.

Webhook Security (NOWPayments IPN)

Every incoming IPN webhook is verified with HMAC-SHA512 before any database write:

// 1. Extract signature from request header
const signature = req.headers['x-nowpayments-sig'];
const ipnSecret = process.env.NOWPAYMENTS_IPN_SECRET;
// 2. Reject immediately if missing
if (!signature || !ipnSecret) { res.status(401); return; }
// 3. Canonicalise body (sorted JSON keys)
const sortedBody = JSON.stringify(
  Object.fromEntries(Object.entries(req.body).sort())
);
// 4. Compute expected signature
const hmac = crypto.createHmac('sha512', ipnSecret).update(sortedBody).digest('hex');
// 5. Reject if mismatch
if (hmac !== signature) { res.status(401); return; }

Authentication & Authorisation

Authentication Flow

POST /api/auth/register
  → Validate email (regex) + password (min 8, 1 uppercase, 1 digit)
  → bcrypt.hash(password, 12)
  → Create User (emailVerified: false)
  → Generate 6-digit OTP via crypto.randomInt(100000, 999999)
  → Send OTP via SMTP
  → 201 { userId, emailVerified: false }

POST /api/auth/verify-email
  → Find user by userId
  → Compare OTP + check expiry (15 min window)
  → Update emailVerified: true, clear OTP fields
  → Sign JWT { userId, memberType, membershipTier }  exp: 7d
  → 200 { token }

POST /api/auth/login
  → Find user by email (lowercase)
  → Check isActive flag
  → bcrypt.compare(password, passwordHash)
  → Check emailVerified (required for MEMBER type)
  → Sign JWT
  → 200 { token, user }

Client stores token in:
  Cookies.set('gc_token', token, { expires: 7, secure: true, sameSite: 'strict' })

JWT Payload

{
  "userId":         "clxxxxx",
  "memberType":    "MEMBER | COMPANION | ADMIN | SUPER_ADMIN",
  "membershipTier": "FREE | SILVER | GOLD | PLATINUM",
  "iat":           1713000000,
  "exp":           1713604800   // 7 days
}

Auth Middleware

The authenticate middleware extracts the Bearer token from the Authorization header, verifies it with jwt.verify(), and attaches req.userId and req.memberType to the request. The requireSuperAdmin middleware checks req.memberType === 'SUPER_ADMIN' and returns 403 otherwise.

Payment Flow

1. Member submits booking → POST /api/bookings
   → Booking created with status: PENDING_PAYMENT
   → Returns bookingId

2. Member clicks "Pay Now" → POST /api/bookings/:bookingId/payment
   → Backend calls NOWPayments Invoice API:
     POST https://api.nowpayments.io/v1/invoice
     {
       price_amount: booking.totalAmount,
       price_currency: "usd",
       order_id: booking.id,
       ipn_callback_url: "https://gild3d.com/api/bookings/webhook/nowpayments",
       success_url: "https://gild3d.com/bookings?ref=...&status=success",
       cancel_url:  "https://gild3d.com/bookings"
     }
   → Payment record upserted (handles retry scenario)
   → Returns { checkoutUrl, invoiceId }

3. Browser redirects to NOWPayments hosted checkout
   → User selects crypto, sends payment

4. NOWPayments calls IPN webhook on payment_status changes:
   waiting → confirming → confirmed → sending → finished

5. On status = "finished":
   → HMAC-SHA512 signature verified
   → Payment.status → CONFIRMED
   → Booking.status → CONFIRMED
   → Confirmation email sent to member via SMTP

Idempotency Note

The payment creation endpoint uses prisma.payment.upsert() keyed on bookingId. If a member clicks "Pay Now" multiple times (e.g. after a failed attempt), the existing Payment record is updated rather than creating a duplicate, preventing unique-constraint errors.

Email Failure Handling

Registration email failures are logged but do not block account creation — the user ID is returned and the member can request a resend. Booking confirmation email failures are caught with .catch(() => {}) to prevent webhook processing failure affecting the payment confirmation state.
