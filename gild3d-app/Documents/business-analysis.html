01   Project Background

This Business Analysis Document (BAD) captures the functional requirements, stakeholder needs, process flows, user stories and acceptance criteria for the Gild3d luxury companion platform. It serves as the formal record of what was specified, what was delivered and what forms the baseline for future development phases.

The analysis was conducted iteratively alongside development, with requirements refined based on technical feasibility and stakeholder feedback. All requirements marked as Delivered are implemented in the current production system.

02   Stakeholder Analysis

Super Administrator

SUPER_ADMIN role  ·  Platform owner / operator
Goals
Full control of platform. Manage companions, oversee all members, audit activity, manage revenue and ensure quality standards are met.
Pain Points
Needs efficient companion onboarding tools; requires oversight of bookings and payments without being involved in every transaction.
Key Interactions
Admin dashboard, companion CRUD, member management, booking/message audit, reports.

Administrator

ADMIN role  ·  Operational staff
Goals
Day-to-day platform operations — approve/reject content, assist members, ensure companion profiles are current and accurate.
Pain Points
Needs clear workflows for content moderation; should not require technical knowledge to perform admin tasks.
Key Interactions
Admin dashboard (subset), profile publishing, member status management.

Member (Client)

MEMBER role  ·  Paying client
Goals
Discover and connect with companions. Browse profiles, make bookings, pay securely and manage their own engagement history.
Pain Points
Privacy is paramount; does not want payment traces. Expects luxury UX — not generic or low-quality interfaces.
Key Interactions
Registration, profile browsing, booking, payment, messaging, membership upgrade.

Companion

COMPANION role  ·  Service provider
Goals
Present a professional profile, receive bookings from quality clients and manage their schedule. Requires platform credibility to attract high-value members.
Pain Points
Profiles managed by admin — companion has no self-service portal in Phase 1. Photo upload and profile editing requires admin mediation.
Key Interactions
Profile (admin-managed), bookings assigned to them, messaging received.

05   User Stories

Member (Client) Stories

Member
As a new member, I want to register with my email and verify via OTP, so that my account is confirmed and my identity is protected.
Member
As a member, I want to browse companion profiles with photos, bios and rates, so that I can find a companion that suits my preferences.
Member
As a member, I want to book a companion for a specific date, duration and location type, so that I can plan an engagement in advance.
Member
As a member, I want to pay for my booking via cryptocurrency, so that my payment is private and leaves no traceable financial record.
Member
As a member, I want to view my booking history and status, so that I can track my engagements and payment confirmations.
Member
As a member, I want to upgrade my membership tier via crypto payment, so that I can unlock access to premium companions and private content.
Member
As a member, I want to cancel a booking before it is confirmed, so that I can change my plans without losing my payment if it has not been processed.

Super Administrator Stories

Super Admin
As a super admin, I want to add a new companion profile with all details and photos, so that I can grow the platform's companion supply.
Super Admin
As a super admin, I want to publish or unpublish a companion profile, so that I control what is visible to members at any time.
Super Admin
As a super admin, I want to edit a companion's hourly rate, age and key details, so that profiles remain accurate and commercially current.
Super Admin
As a super admin, I want to delete a companion or member account, so that I can remove platform participants who violate terms or are no longer active.
Super Admin
As a super admin, I want to view all bookings with status and amounts, so that I can audit platform activity and ensure revenue accuracy.
Super Admin
As a super admin, I want to see platform KPIs on a dashboard, so that I have visibility of member growth, booking volumes and revenue at a glance.

07   Assumptions & Constraints

Assumptions

#AssumptionImpact if Wrong
A1NOWPayments API remains available and supported throughout platform lifecyclePayment system would require migration to alternative crypto processor
A2All companion profiles are created and managed by administrators (no self-service in Phase 1)If companions need self-service, a companion portal is required in Phase 2
A3Platform operates from a single geographic region initially (AWS Lightsail, single AZ)Regional failure would cause platform downtime; multi-AZ required for HA at scale
A4Members are adults (18+) in jurisdictions where platform use is legalAge verification and geo-restriction features must be implemented before operating in regulated markets
A5Cryptocurrency adoption among target member demographic is sufficient for payment conversionFiat payment fallback (Stripe, bank transfer) would need to be added in Phase 2
A6Email delivery via Nodemailer/SMTP is reliable for OTP and notificationsTransactional email failure impacts registration; managed provider (SendGrid, Postmark) recommended

Constraints

#ConstraintRationale
C1All payment channels must be cryptocurrency onlyBusiness decision for privacy. Fiat processors may decline the merchant category.
C2Companion profiles must be admin-created, not self-serviceQuality and legal control requirement. Prevents unvetted content appearing on platform.
C3Platform must operate over HTTPS at all timesSecurity and trust requirement. HTTP access is redirected.
C4File uploads are restricted to images ≤ 20MB, validated by MIME type and extensionSecurity constraint to prevent malicious file uploads.
C5Session tokens are not accessible via JavaScript (HttpOnly cookie)XSS protection — tokens cannot be exfiltrated by scripts.
08   Gap Analysis & Future Requirements

The following items were identified during analysis but are deferred to Phase 2 or beyond, due to scope, complexity or sequencing:

Phase 2

Real-Time Messaging (WebSocket)

Current messaging is stored but not real-time. WebSocket or Socket.io integration required for live chat experience between members and companions.

Phase 2

Companion Self-Service Portal

Companions currently rely on admins for all profile changes. A self-service portal with admin approval workflow would reduce operational burden and improve companion experience.

Phase 2

Companion Availability Calendar

No availability management exists in Phase 1. A calendar-based availability system would prevent double-booking and improve booking conversion.

Phase 2

Booking Email Notifications

Automated email notifications for booking confirmation, cancellation and payment confirmation are not yet implemented. Required for professional member experience.

Phase 2

Companion Rating & Review System

No feedback mechanism exists. A moderated review system would build trust and surface quality companions to members.

Phase 2

Password Reset Flow

Members who forget their password currently have no self-service recovery. Email-based reset link required.

Phase 2

GDPR / Privacy Rights Portal

Members should be able to request data export and account deletion. Currently admin-mediated only. Regulatory requirement in EU/UK markets.

Phase 3

Native Mobile Applications

iOS and Android apps would significantly increase member engagement and retention. React Native or Flutter recommended for code reuse with existing API.

Phase 3

Multi-Region Infrastructure

Current single-region AWS Lightsail deployment has no geographic redundancy. As platform scales, ECS + RDS Multi-AZ + CloudFront recommended.

09   Phase 1 Acceptance Criteria

The following criteria define the acceptance baseline for Phase 1 sign-off:

  • A new user can register, receive and validate an OTP, and access the platform within a single session
  • A logged-in member can browse all published companion profiles and view photos
  • A member can create a booking specifying date, type (incall/outcall), hours and companion
  • A member can initiate a crypto payment and receive a NOWPayments checkout URL
  • A booking status transitions from PENDING_PAYMENT to CONFIRMED upon payment webhook receipt
  • A Super Admin can log in and access the admin dashboard with companion and member data
  • A Super Admin can add a new companion profile with all fields and at least one photo
  • A Super Admin can publish and unpublish a companion profile
  • A Super Admin can edit companion rate and age; changes appear immediately on the profile
  • A Super Admin can delete a companion or member account with inline confirmation
  • All API endpoints return appropriate HTTP status codes and JSON error messages
  • All pages load over HTTPS; HTTP requests are permanently redirected (301)
  • Rate limiting blocks login attempts exceeding 5 per 15 minutes from the same IP
  • File uploads are rejected for non-image MIME types and files exceeding 20MB
  • The platform loads without JavaScript errors in Chrome, Firefox and Safari (latest versions)
