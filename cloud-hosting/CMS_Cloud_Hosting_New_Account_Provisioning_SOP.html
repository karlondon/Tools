1. Overview & Scope

This SOP describes the end-to-end process for receiving, validating, and fulfilling new Cloud Hosting account provisioning requests managed by the CMS team. It covers all three cloud providers (GCP, AWS, Azure) and applies to all Service Operations staff handling cloud provisioning tickets.

📌 Before You Begin Always direct the requester to the Cloud Hosting site first. It contains account type descriptions, the inquiry checklist, compliance process links, and the pricing model. This single action prevents the majority of back-and-forth on tickets.

2. GCP Organisation Types

GCP has two distinct organisation environments. Determining the correct one is a critical early step.

Organisation Description Who Manages It When to Use
NSE Org / Internal Org Internal Deloitte organisation (also referred to as NSE Org or Internal Org). Tickets assigned here go straight to the Platforms team – CMS cannot action them directly. Platforms Team Deloitte-only use; no client access to the environment; Deloitte-managed data (even if confidential/IP); internal assets or products being built to sell.
Resell Org (External) External organisation where CMS runs the pipeline to provision accounts. CMS (via pipeline) Client will log in or access the environment; client manages any part of the data; any data classified as client-owned regardless of environment type.
💡 Rule of Thumb – NSE/Internal Org vs Resell (GCP)
  • If only Deloitte manages and accesses the environment → NSE / Internal Org
  • If the client will access the environment (even read-only, via web/app/mobile) → Resell Org
  • If unsure, reach out to Rivin Jose Elavathingal or the Platform team for data classification guidance.
📌 Azure & AWS Note Azure provisioning is always internal/MSE-side – the Platforms team handles it. AWS has similar internal/external concepts; confirm with the team lead if unsure.

3. Account / Environment Types

Type Description PII / Client Data Allowed? GTOM Required?
Sandbox Exploratory, experimental environment No No
Non-Production Development, testing, staging environments No – dummy/public data only Recommended to start
Production Live environment with real or client data Yes – but must be Production account Yes – mandatory, can take months
⚠️ Critical Rule If the account will contain any Personally Identifiable Information (PII) or real client data, it must be provisioned as a Production account. A non-prod account must never hold PII. If the requester hasn't started GTOM, advise them to do so immediately – GTOM for Production can take several months.

4. End-to-End Process Flow

Each numbered step below corresponds to a stage in the provisioning lifecycle.

1
📥 Ticket Received – Initial Inquiry

A new cloud hosting request arrives via the Opportunities channel in ServiceNow. The requester has used the "Inquire Now" button on the Cloud Hosting site and submitted basic information: project name, description, project type, readiness level, compliance status, business function, cloud provider, and environment type.

📌 Note on the Current ServiceNow Form Limitations The current form only captures a single dropdown for "Deloitte Compliance" (Started / Not Started). It does not ask for GTOM, InfoSec, or NSE Catalyst status individually. The Platform team is working on improving this form. Always triage carefully.
2
🔗 Send Cloud Hosting Site Link to Requester

Before asking any questions or chasing compliance details, send the requester the Cloud Hosting site link. This contains:
– Account type descriptions (Sandbox / Non-Prod / Production)
– Pre-inquiry checklist (WBS code, partners on board, compliance processes started)
– Links to GTOM, InfoSec, and NSE Catalyst processes
– Pricing model overview

💡 Why this matters Previously, teams asked individually about GTOM, InfoSec, and AI compliance via ticket messages, causing weeks of back-and-forth. Directing to the site upfront saves significant time for both sides.
3
🔍 Triage & Validate the Request

Review the ticket and determine:

✅ Sufficient Info? Proceed to compliance and org-type assessment (Step 4).
❓ Missing Info? Respond via ServiceNow ticket requesting missing details. If no response, arrange a Teams call (15–30 min) to gather requirements. Remind them to document responses in the ticket.

Key questions to validate: Does the account contain PII? Will the client access it? What is the data classification? Is there a WBS code? Has GTOM/InfoSec/NSE Catalyst been started? Which cloud provider and environment type?

💡 Tip – Requesters and Teams Messages Some requesters will proactively message the assigned person on Teams. Ask them to put all queries into the ServiceNow ticket to maintain an audit trail.
4
⚖️ Determine Organisation Type & Compliance Route

Based on the validated information, determine the correct org type and compliance path:

🏢 NSE / Internal Org (GCP/AWS) Deloitte-only access, no client-facing usage. Proceed to create P-code and raise with Platforms team.
🌐 Resell Org (GCP) Client will access the environment. Pipeline provisioning required. Ensure GTOM and InfoSec are complete before proceeding.
🔷 Azure Always Internal/MSE. Choose Hub Connected (global Deloitte AD access) or Hub Disconnected (isolated, manually managed users). Platforms team provisions.
5
📂 Create P-Code in ServiceNow

Create a new Project Code (P-Code) in ServiceNow. This is required before raising a platform request – the platform request system will not link without a valid P-code.

When creating the P-code, the initial status will be set to "Opportunity".

⚠️ Critical – Change P-Code Status to "Build" After creating the P-code, you must change its status from Opportunity → Build. If it remains in "Opportunity" status, the platform request form will not be able to link to it and provisioning cannot proceed. This is a commonly missed step.
💡 P-Code Generation Tip The follow-up survey in ServiceNow can auto-generate a P-code with pre-filled information. If you need to manually find the next available P-code number, use the search filter P000 to list recent codes and identify the next sequential one.
6
📧 Send Terms & Conditions Approval Email to Partner/Director

Before raising the account request with the Platform team, send a Terms & Conditions approval email to the responsible Partner or Director (whoever is approving the account creation).

The email must include:

  • Cloud service provider (e.g. Google Cloud Platform / Azure / AWS)
  • Description of the solution/workload
  • WBS code to be used
  • Chargeable confirmation
  • Management tier (Self-Managed / Co-Managed / Fully Managed)
The Partner/Director must explicitly reply with approval wording. Keep this email as evidence – it can be attached to the platform request.

⚠️ No approval = No provisioning Do not raise the platform request without documented partner/director approval. Store the approval email for audit purposes.
7
🖥️ Raise Cloud Account Request with Platform Team

Using the internal platform request link, raise the new cloud account request. Ensure all required fields are completed before submitting. See Section 5 for the full field reference.

Attach the partner approval email to the request before submitting.

📌 Recommended Before submitting a real request, browse the form without submitting to familiarise yourself with all available fields and dropdowns. This helps you prepare the right questions to ask requesters in advance.
8
⏱️ Platform Team Provisions the Account (SLA: 3 Working Days per Account)

Once submitted, the Platform team will pick up the request and provision the account. The SLA is 3 working days per account.

Important: Accounts are provisioned sequentially, not in parallel. If multiple accounts are requested (e.g. non-prod + prod + dev), each takes 3 working days separately.

⚠️ Date Submission Rules
  • Always request provisioning on a Monday or working day.
  • If a Monday is a Bank Holiday, select the next working day.
  • The Platform team will reject requests dated on non-working days and ask you to resubmit.
9
✅ Account Provisioned – Confirm & Close

Once the Platform team confirms provisioning:

  • Notify the requester via ServiceNow ticket and/or Teams
  • Confirm account details and access instructions
  • Update the ServiceNow ticket to reflect completion
  • Close the ticket

5. OpenCloud Insights Portal – Client Inquiry Process

The OpenCloud Insights Portal is the client-facing portal used to kick-start the Cloud Hosting enquiry process. Before a ticket even reaches the CMS team, clients (or internal requesters) interact with this portal. Understanding its layout helps the team triage incoming requests effectively.

📌 Portal Access The OpenCloud Insights Portal is accessed at https://eu.opencloudportal.com/. The portal has several sections relevant to the provisioning process:
  • Home / Overview – Landing page with navigation to all portal sections
  • Cloud Financials – https://eu.opencloudportal.com/cloud-financials – Cost visibility and WBS tracking
  • Security – https://eu.opencloudportal.com/security – Compliance and security posture
  • Admin – https://eu.opencloudportal.com/admin – Administrative functions

How Clients Raise an Inquiry

The client inquiry journey through the Cloud Hosting site follows this sequence before a ServiceNow ticket is created:

StageWhat the Client / Requester DoesCMS Team Note
1. Visit Cloud Hosting Site Requester visits the internal Cloud Hosting information page. Previously, an "Inquire Now" button was prominently placed at the top — this has been moved lower to encourage reading the content first. The Cloud Hosting site now contains the full inquiry checklist, account type descriptions, compliance links, and pricing model. Always send this link first.
2. Review Account Types Requester reads about Sandbox, Non-Production, and Production account types to determine what they need. If they have PII or client data, they must choose Production. Flag this early to avoid delays.
3. Complete Pre-Inquiry Checklist Requester confirms they have: partners on board, WBS code arranged, and relevant compliance processes (GTOM / InfoSec / NSE Catalyst) started. This checklist is designed to reduce back-and-forth. If they haven't completed it, send them back to the site before progressing.
4. Click "Inquire Now" Requester scrolls to the bottom of the Cloud Hosting site and clicks the Inquire Now button to raise a ServiceNow ticket. The ServiceNow form captures: project name, description, project type, readiness, compliance status (single dropdown only), business function, cloud provider, environment type.
5. Ticket Lands in Opportunities Channel — CMS team receives the ticket in the ServiceNow Opportunities channel. Triage begins from here (see Process Flow, Step 3).

8. Tips, Notes & Gotchas

Compliance Processes

💡 GTOM (Go To Market) Required for all Production accounts and mandatory before a Production account can be provisioned. GTOM can take months for production workloads. Advise requesters to start GTOM as early as possible. Do not wait for the ticket to be raised before starting GTOM.
💡 InfoSec Review Must be started and a ticket reference obtained before provisioning. Include the InfoSec ticket number in the Additional Information field on the request form.
💡 NSE Catalyst Another compliance process required. Obtain the reference number and include in the request.

P-Code Tips

📌 P-Code Auto-Generation The ServiceNow follow-up survey can auto-generate a P-code with information pre-filled. Use this where possible to save time. If the follow-up survey is not relevant to your provider, manually identify the next available P-code using the P000 search filter.
⚠️ P-Code Status Must Be "Build" Always verify the P-code status is set to Build before raising the platform request. Searching by P-code (e.g. P000639) will show you the current status. If it shows "Opportunity", update it to "Build" first. The platform form will silently fail to link if the status is wrong.

WBS Code

💡 WBS Code Can Cause Delays Obtaining a valid WBS code can take significant time. Chase this early. The WBS code must be approved by a Partner/Director before the T&C email can be sent.

Provisioning SLA & Multi-Account Requests

⚠️ 3 Working Days Per Account – Not Parallel Each account takes 3 working days to provision. Multiple accounts (e.g. non-prod + prod + dev) are provisioned sequentially, not in parallel. Set realistic expectations with requesters at the outset.

Communication & Escalation

💡 Requesters Not Responding on Tickets? Set up a brief Teams call (15–30 minutes) to gather all required information at once. Remind requesters to document all decisions in the ServiceNow ticket for audit purposes. Some requesters will proactively message assignees on Teams – redirect them to the ticket.

Environment Type Mismatches

💡 ServiceNow vs. Real Environment Names ServiceNow's environment dropdown does not always match the environments teams use (e.g. "Dev" may not appear). Select the closest option available (Non-Prod or Dev) and update the ticket/record after provisioning if needed.

Azure Teams Note

📌 Azure Provisioning Azure provisioning is straightforward once all information is gathered – it is always internal (MSE-side). Once you have all required details, pass directly to the Platforms team. Pull in Karthik for any complex Azure queries.
